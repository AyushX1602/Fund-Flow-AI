================================================================================

  Browser

    ↓

    │

    ├─[5] rateLimiter          check 100 req/15min per IP
    ├─[6] auth middleware
    │      ↳ header X-Demo-Mode: true?

    │           NO  → verify JWT → extract userId + role
    ├─[7] validate(schema)

    │      ↳ invalid → 400 { success:false, message:"..." }
    │
    ├─[8] Route matches  POST /transactions
    │      ↳ transactionController.createTransaction(req, res)
    │
    ├─[9] Controller calls Service
    │      ↳ transactionService.processTransaction(data, userId)
    │
    ├─[10] Service calls Prisma
    │      ↳ prisma.transaction.create({...})
    │      ↳ Prisma generates SQL → PostgreSQL
    │
    ├─[11] Controller builds response
    │      ↳ ApiResponse.success(res, 201, "Transaction processed", data)
    │
    └─[12] Response
           { success: true, message: "...", data: { transaction, mlResult, alert } }
           ↓
         Browser axios interceptor
           response => response.data   ← unwraps envelope automatically

DIAGRAM 3 — TRANSACTION SUBMISSION FULL PIPELINE
================================================================================

  User on AnalyzePage clicks "Run Full Analysis"
    │
    ├── activeScenario is SET (preset clicked)?
    │     YES → return DEMO_RESULTS[activeScenario]  after 1.2s fake delay
    │            No API call. No DB write.
    │
    └── NO → real API path continues below
          │
          POST /api/transactions
          │
          transactionController.createTransaction
          │
          transactionService.processTransaction(data, userId, opts)
          │
          ├─[A] Fetch senderAccount from DB
          │     prisma.account.findUnique({ where: { id: data.senderAccountId } })
          │
          ├─[B] Fetch receiverAccount from DB
          │
          ├─[C] Parse UPI VPA
          │     "user123@oksbi" → pspSender = "SBI"
          │     "merchant99@ybl" → pspReceiver = "YES_BANK"
          │
          ├─[D] Create Transaction record
          │     prisma.transaction.create({
          │       transactionId, amount, currency, type, channel,
          │       senderAccountId, receiverAccountId,
          │       upiVpaSender, upiVpaReceiver, pspSender, pspReceiver,
          │       vpaAgeDays, chainId, ringId, ipAddress,
          │       location, description, timestamp
          │     })
          │
          ├─[E] Socket.io emit TRANSACTION_NEW
          │     → browser LiveTransactionFeed updates
          │
          ├─[F] mlService.scoreTransaction()   ← RULE-BASED ENGINE
          │     (XGBoost NOT used for live scoring — see note in Diagram 4)
          │
          │     SCORING RULES (additive, clamped 0.0–1.0):
          │     ─────────────────────────────────────────
          │     amount > 500,000                    → +0.30
          │     UPI + amount > 50,000               → +0.25
          │     45,000 ≤ amount ≤ 50,000            → +0.25  (structuring)
          │     900,000 ≤ amount ≤ 1,000,000        → +0.30  (PMLA threshold)
          │     kycType == OTP_BASED                → +0.10
          │     kycType == MIN_KYC                  → +0.15
          │     sender.kycFlagged == true           → +0.20
          │     sender.muleScore > 0.5              → +0.20
          │     receiver.muleScore > 0.5            → +0.15
          │     sender.bankName ≠ receiver.bankName → +0.05
          │     account age < 30 days               → +0.15
          │     vpaAgeDays < 7                      → +0.10
          │     sender.isFrozen                     → +0.40
          │     receiver.isFrozen                   → +0.30
          │     hour ∈ [1, 5] (1AM–5AM)             → +0.10
          │     description has scam keyword        → +0.35
          │       keywords: prize, lottery, investment return,
          │                 forex profit, job advance, loan, otp,
          │                 urgent help, award money, kyc update
          │
          │     fraudScore = clamp(sum, 0, 1)
          │     isFraud    = fraudScore >= config.alertThreshold (0.70)
          │     return { fraudScore, isFraud, reasons[], modelVersion }
          │
          ├─[G] Promise.all — THREE parallel tasks:
          │
          │     [G1] prisma.transaction.update
          │           fraudScore, isFraud, mlModelVersion, mlReasons JSON
          │
          │     [G2] prisma.fundFlowEdge.create
          │           sourceAccountId → targetAccountId
          │           amount, riskScore, transactionId, timestamp
          │
          │     [G3] riskEngine.computeRiskLayers()   ← 6-LAYER ENGINE
          │          ┌───────────────────────────────────────────────┐
          │          │ L1 Location (weight 12%)                      │
          │          │   bankName has "cooperative/urban/gramin"?    │
          │          │   city in TIER3_CITIES (Bhagalpur, Gorakhpur)?│
          │          │   PSP in HIGH_RISK_PSPS (fino,paytm,airtel)?  │
          │          │                                               │
          │          │ L2 Channel (weight 10%)                       │
          │          │   MOBILE_APP=0.25  NET_BANKING=0.15           │
          │          │   ATM=0.40         BRANCH=0.10                │
          │          │                                               │
          │          │ L3 Behavioral (weight 18%)                    │
          │          │   amount vs sender 30-day avg from DB         │
          │          │   deviation > 3× avg → high score             │
          │          │                                               │
          │          │ L4 ML Score (weight 35%)                      │
          │          │   = fraudScore from rule engine above         │
          │          │                                               │
          │          │ L5 Network Graph (weight 15%)                 │
          │          │   ring membership via FundFlowEdge DB queries │
          │          │   chainId matches → elevated score            │
          │          │                                               │
          │          │ L6 Velocity (weight 10%)                      │
          │          │   count transactions in last 5 min for sender │
          │          │                                               │
          │          │ compositeScore =                              │
          │          │   (0.7 × max_layer) + (0.3 × weighted_avg)   │
          │          └───────────────────────────────────────────────┘
          │
          ├─[H] Socket.io emit TRANSACTION_SCORED
          │
          ├─[I] Gemini / Ollama LLM  (see Diagram 7)
          │     effectiveScore = max(fraudScore, compositeScore)
          │     ONLY if 0.35 ≤ effectiveScore ≤ 0.75  (uncertain zone)
          │     Otherwise llmAnalysis = null
          │
          ├─[J] Alert creation
          │     if isFraud OR effectiveScore >= 0.70:
          │       alertType = from reasons:
          │         MULE_ACCOUNT / SUSPICIOUS_PATTERN /
          │         HIGH_VALUE / VELOCITY_BREACH / FRAUD_DETECTED
          │       severity = score >= 0.80 → CRITICAL
          │                  score >= 0.60 → HIGH
          │                  score >= 0.30 → MEDIUM  else LOW
          │       prisma.alert.create({
          │         alertType, severity, transactionId,
          │         description, riskScore,
          │         mlReasons: {                ← JSON blob
          │           reasons, mlScore, compositeScore,
          │           layers, dominantLayer,
          │           llm: { verdict, confidence, reasoning, flags }
          │         }
          │       })
          │       Socket.io emit ALERT_CREATED
          │
          ├─[K] Update account risk scores (sender + receiver)
          │     last 50 transactions, weighted decay: weight = 1/(idx+1)
          │     riskScore = weighted avg of fraudScores
          │     muleScore = fraudAsSender / total × 2, capped at 1
          │
          └─[L] Return HTTP 201
                { success, data: { transaction, mlResult, riskProfile,
                                   llmAnalysis, alert } }

    │     X = np.array([[feature_values]])
    │     raw_prob = xgboost_model.predict_proba(X)[0][1]
    │
    ├─[D] Platt Scaling Calibration
    │     A = -92.0,  B = 88.0
    │     calibrated = 1 / (1 + exp(A × raw_prob + B))

    │       raw=0.87 → calibrated ≈ 0.00  (safe)
    │       raw=0.94 → calibrated ≈ 0.18  (low)

    │     "contribution" = importance value (not a log-odds contribution)
    │
    └─[G] Return
          { fraud_score, is_fraud, reasons[], risk_tier,
            model_version:"xgboost-v1", raw_model_prob }

  !! IMPORTANT NOTE !!
  Node.js does NOT call FastAPI for live transaction scoring.
  FastAPI is only called by:

  The rule-based engine (Diagram 3-F) handles all live scoring.
  Reason: XGBoost outputs 0.87–0.99 for all transactions without
  real account history, making calibration unreliable.

DIAGRAM 7 — LLM ROUTING LOGIC
================================================================================

  llmService.analyseTransaction(transaction, senderAccount, mlResult, riskProfile)
    │
    effectiveScore = max(mlResult.fraudScore, riskProfile.compositeScore)
    │
    ├── effectiveScore < 0.35  → return null  (clearly legit, skip LLM)
    ├── effectiveScore > 0.75  → return null  (clearly fraud, skip LLM)
    └── 0.35 ≤ score ≤ 0.75   → continue below
          │
          cacheKey = hash of (transactionId + top features)
          │
          ├── analysisCache.has(cacheKey) AND age < 30min?
          │     YES → return cached verdict  (fromCache: true)
          │     NO  → continue
          │
          LLM_PROVIDER env var:
          ├── "ollama" → go direct to Ollama
          ├── "gemini" → go direct to Gemini
          └── "auto"   → try Ollama first (DEFAULT)
                │
                isOllamaRunning()?
                  GET http://localhost:11434/api/tags
                  check model list includes "qwen3"
                │
                ├── YES → callOllama()
                │         POST :11434/api/generate
                │         model: "qwen3:8b"
                │         keep_alive: "5m"    ← stays in VRAM
                │         stream: false
                │         timeout: 60s
                │         prompt: structured fraud context + question
                │
                └── NO  → try Gemini
                          GEMINI_API_KEY in .env?
                          │
                          ├── NO KEY  → return null
                          │
                          └── YES
                                dailyCallCount >= 100? → return null
                                Date.now() - lastCallTime < 4000?
                                  → wait remainder  (15 RPM guard)
                                dailyCallCount++
                                lastCallTime = Date.now()
                                │
                                geminiModel.generateContent(prompt)
                                timeout: 25s
                                model: gemini-2.5-flash-preview-04-17

          Parse JSON response → extract:
            verdict:    LIKELY_FRAUD | LIKELY_LEGITIMATE |
                        UNCERTAIN | NEEDS_INVESTIGATION
            confidence: 0.0–1.0
            reasoning:  free text explanation
            flags:      string array of triggered risk signals

          Store in analysisCache (30min TTL)
          Return { verdict, confidence, reasoning, flags, fromCache }

DIAGRAM 8 — DATABASE SCHEMA (PostgreSQL via Prisma)
================================================================================

  User
  ├── id          String  PK (cuid)
  ├── email       String  UNIQUE
  ├── passwordHash String
  ├── firstName   String
  ├── lastName    String
  ├── role        Enum    ADMIN | ANALYST | VIEWER
  └── createdAt   DateTime

  Account
  ├── id              String   PK
  ├── accountNumber   String   UNIQUE
  ├── accountHolder   String
  ├── bankName        String
  ├── bankBranch      String?
  ├── accountType     Enum     SAVINGS | CURRENT | ...
  ├── kycType         Enum     FULL_KYC | MIN_KYC | OTP_BASED
  ├── kycFlagged      Boolean
  ├── kycFlagReason   String?
  ├── muleScore       Float    (0.0–1.0, rolling updated)
  ├── riskScore       Float    (0.0–1.0, rolling updated)
  ├── isFrozen        Boolean
  ├── frozenAt        DateTime?
  ├── frozenReason    String?
  └── createdAt       DateTime

  Transaction
  ├── id              String  PK
  ├── transactionId   String  UNIQUE  (TXN-XXXXX format)
  ├── amount          Float
  ├── currency        String  default "INR"
  ├── type            Enum    UPI | NEFT | IMPS | ATM | DEPOSIT
  ├── channel         Enum    MOBILE_APP | NET_BANKING | ATM | BRANCH
  ├── senderAccountId String  FK → Account
  ├── receiverAccountId String FK → Account
  ├── fraudScore      Float?  (null until scored)
  ├── isFraud         Boolean default false
  ├── mlReasons       Json?   (array of {feature, impact, value, description})
  ├── mlModelVersion  String?
  ├── upiVpaSender    String?
  ├── upiVpaReceiver  String?
  ├── pspSender       String?
  ├── pspReceiver     String?
  ├── vpaAgeDays      Int?
  ├── chainId         String?
  ├── ringId          String?
  ├── ipAddress       String?
  ├── deviceId        String?
  ├── location        String?
  ├── description     String?
  └── timestamp       DateTime

  Alert
  ├── id            String  PK
  ├── alertType     Enum    FRAUD_DETECTED | MULE_ACCOUNT |
  │                         SUSPICIOUS_PATTERN | HIGH_VALUE |
  │                         VELOCITY_BREACH
  ├── severity      Enum    LOW | MEDIUM | HIGH | CRITICAL
  ├── status        Enum    NEW | INVESTIGATING | ESCALATED |
  │                         RESOLVED_FRAUD | RESOLVED_LEGITIMATE | DISMISSED
  ├── transactionId String  FK → Transaction  UNIQUE
  ├── riskScore     Float
  ├── description   String
  └── mlReasons     Json    {

  Investigation
  ├── id          String  PK
  ├── alertId     String  FK → Alert  UNIQUE
  ├── title       String
  ├── status      Enum    OPEN | IN_PROGRESS | CLOSED | ESCALATED
  ├── priority    Enum    LOW | MEDIUM | HIGH | CRITICAL
  ├── assignedTo  String? FK → User
  └── notes       CaseNote[]

  CaseNote
  ├── id              String  PK
  ├── investigationId String  FK → Investigation
  ├── userId          String  FK → User
  ├── content         String
  └── noteType        Enum    OBSERVATION | ACTION | ESCALATION | RESOLUTION

  FundFlowEdge
  ├── id              String  PK
  ├── chainId         String?
  ├── sourceAccountId String  FK → Account
