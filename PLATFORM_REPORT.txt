--------------------------------------------------------------------------------
1. WHAT THIS SYSTEM IS
--------------------------------------------------------------------------------

FundFlow AI is a real-time fraud detection and investigation platform built
for Public Sector Banks (PSBs) in India. It processes UPI, NEFT, IMPS, and
ATM transactions, assigns risk scores, and flags potential fraud using a
multi-layered scoring system.

It is NOT a production-grade deployed system. It is a working hackathon
prototype with a real ML pipeline, real DB, and real-time WebSocket feeds.

4. SCORING SYSTEM — HOW IT ACTUALLY WORKS
--------------------------------------------------------------------------------

There are THREE scoring components. Not all run for every transaction.

COMPONENT 1: Rule-Based Engine (mlService.js → ruleBasedScore())
  This is what runs for EVERY live transaction submitted via the Analyze page
  and API. XGBoost is NOT used for live transaction scoring (see why below).

  Rules and weights:
    amount > 500,000            +0.30
    UPI > 50,000                +0.25
    45,000 ≤ amount ≤ 50,000   +0.25  (near ₹50K reporting threshold)
    900,000 ≤ amount ≤ 1M      +0.30  (near ₹10L PMLA threshold)
    KYC = OTP_BASED             +0.10
    KYC = MIN_KYC               +0.15
    sender KYC flagged          +0.20
    sender muleScore > 0.5     +0.20
    receiver muleScore > 0.5   +0.15
    cross-bank transfer         +0.05
    account age < 30 days       +0.15
    VPA age < 7 days            +0.10
    sender account frozen       +0.40
    receiver account frozen     +0.30
    transaction hour 1AM-5AM   +0.10
    scam keywords in remarks    +0.35
      (keywords: prize, lottery, investment return, forex profit, job advance,
       loan, otp, urgent help, award money, kyc update)

  Score clamped to [0.0, 1.0]. isFraud = score >= alertThreshold (default 0.70).
  modelVersion returned: "rule-based-v1"

COMPONENT 2: XGBoost Model (FastAPI, predictor.py)
  Trained on PaySim dataset (~500K synthetic transactions).
  49 features including:
    - amount_log, amount_bucket
    - type_UPI, type_NEFT, type_IMPS, type_ATM
    - channel_mobile, channel_internet
    - sender_avg_amount, sender_std_amount, amount_deviation
    - sender_txn_count_1h, sender_txn_count_24h
    - time_since_last_txn_min
    - is_new_receiver, is_cross_bank_upi, upi_new_recv_risk
    - near_50k_threshold, near_100k_threshold, near_1m_threshold
    - is_round_10k, is_round_1k
    - high_velocity_1h, high_velocity_24h, rapid_succession
    - sender_mule_score, sender_in_ring, sender_passthrough_ratio
    - receiver_mule_score, receiver_unique_senders_total
    - is_night, hour_of_day, hour_velocity_ratio
    - sender_is_new_account, receiver_is_new_account
    + India-specific features from india_extras.pkl (VPA, KYC, CIBIL)

  WHY NOT USED FOR LIVE SCORING:
    XGBoost was trained on batch historical data where each account has
    hundreds of transactions (rolling averages, velocity counts). In live
    mode, these features default to neutral approximations (population median
    amounts, neutral times). This causes the model to output raw probabilities
    of 0.87-0.99 for nearly ALL transactions regardless of actual risk.
    Calibration attempts (Platt scaling) failed because the model's output
    distribution differs per transaction type (UPI → 0.94, NEFT → 0.85).
    So XGBoost is kept for the ML Model page metrics display only.

  modelVersion returned: "xgboost-v1"

COMPONENT 3: 6-Layer Composite Engine (riskEngine.js)
  Runs in parallel with rule-based scoring. Uses DB queries.
  Formula: FinalScore = (0.7 × MaxLayer) + (0.3 × WeightedAvg)

  Layers and weights:
    L1 Location Anomaly   weight=0.12
      - High-risk bank keywords (cooperative, urban, rural, gramin)
      - Tier-3 city origin (Bhagalpur, Gorakhpur, etc.)
      - High-risk PSP (fino, paytm, airtel, ibl)
    L2 Channel Risk       weight=0.10
      - MOBILE_APP=0.25, NET_BANKING=0.15, ATM=0.40, BRANCH=0.10
    L3 Behavioral Pattern weight=0.18
      - Amount deviation vs sender's 30-day average
    L4 ML Probability     weight=0.35
      - The rule-based fraudScore (not XGBoost)
    L5 Network Graph      weight=0.15
      - Ring/chain membership via FundFlowEdge DB queries
    L6 Velocity           weight=0.10
      - Transactions in last 5 minutes for sender account

  Composite score stored in Alert.mlReasons.compositeScore
  dominantLayer = whichever layer scored highest

EXPLAINABLE AI:
  - Risk Signals panel: shows rule names + impact values from Component 1
  - "SHAP-lite" labels from XGBoost feature importance (NOT real SHAP)
  - shapLabels.js in frontend: maps raw feature names to human-readable labels
    e.g. "type_UPI" → "UPI Transaction Type"
  - LLM verdict: free-text reasoning from Gemini/qwen3:8b
  - dominantLayer: tells investigator which of the 6 layers triggered alert

7.2 API CONTRACT

  The frontend expects ALL API responses in this envelope:
    { success: boolean, message: string, data: <payload> }

  The axios interceptor (api.js line 14) already unwraps this:
    response => response.data   ← means callers get the envelope, not payload

  IMPORTANT: After await api.post('/transactions', ...) the result is
  the full envelope { success, data: { transaction, mlResult, alert } }.
  Unwrap manually: res.data.transaction (or res.data?.data?.transaction
  depending on interceptor version).

8. KNOWN ISSUES / HONEST LIMITATIONS
--------------------------------------------------------------------------------

1. XGBoost NOT used for live scoring
   See explanation in Section 4. The model runs fine in FastAPI but its
   output can't be reliably calibrated for single-transaction real-time use.
   It is only useful in batch scoring mode (update_db_scores.py).

2. "SHAP" labels are fake
   The feature contribution values shown as "Risk Signals" are:
   feature_importance_value × 1 (just the importance weight, not real SHAP).
   Real SHAP requires the shap library and per-prediction TreeExplainer calls.
   That is NOT implemented.

3. Recharts warning (cosmetic)
   The ChannelDonut.jsx chart previously showed "width(-1) height(-1)" warning.
   Fixed by using explicit 230×230 pixel dimensions instead of "100%".
   If the chart container is hidden on initial mount, you may still see this.

4. No Kafka
   The platform uses synchronous in-process scoring. WebSocket events are
   emitted directly from transactionService.js. No message queue is used.

5. No real account history in ML features
   The fastAPI service computes rolling features (sender_avg_amount etc.)
   from the india_extras.pkl which contains PaySim training data accounts.
   Our seeded accounts (15 accounts) are not in that dataset, so the ML
   features fall back to neutral hard-coded defaults.

6. Database seeding
   Historical data is seeded via node seed-history.js (577 transactions).
   If DB is cleared, the dashboard trend charts will be empty.
   Reseed with: node clear-data.js && node seed-history.js

7. LLM is slow
   Ollama qwen3:8b is an 8B parameter model. First response takes 5-15s.
   Subsequent responses with keep_alive="5m" are 2-5s if model is in VRAM.
   Gemini 2.5 Flash is faster (~1-2s) but has rate limits on free tier.

8. Auth is demo-only
   Real JWT auth is implemented but bypassed via X-Demo-Mode header.
   No session persistence, no refresh tokens.

9. Account risk scores drift
   Each transaction updates sender and receiver account muleScore / riskScore
   via rolling weighted average. If you submit many fraud scenarios, accounts
   will accumulate high mule scores and start flagging legitimate transactions.
   Reset with: node server/reset-accounts.js
